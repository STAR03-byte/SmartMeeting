# SmartMeeting V0.2 协作能力增强设计

**日期**: 2026-03-17  
**状态**: Draft  
**项目**: SmartMeeting  
**基线提交**: `f49371c`

## 1. 背景

当前 SmartMeeting 已完成 MVP 主链路：会议管理、音频上传、转写、纪要生成、任务提取、任务状态流转，以及基础前端联调。

现阶段的主要短板不是 AI 能力缺失，而是“会议结果如何被分享、查找与持续跟进”仍不完整。根据立项书的 V0.2 路线，下一步应优先补齐轻量协作能力，而非直接引入完整鉴权体系或继续增强大模型能力。

## 2. 目标

本阶段聚焦三个用户可感知能力：

1. 会议结果可分享
2. 会议与任务可快速搜索 / 筛选
3. 任务具备基础到期提醒感知

目标是把当前“可演示的单人版”推进到“可在小团队内试用的协作版”。

## 3. 范围

### In Scope

- 为会议生成基础只读分享链接
- 增加公开只读会议详情访问能力
- 会议列表支持按标题 / 状态搜索筛选
- 任务列表支持按状态 / 优先级 / 会议筛选
- 任务增加即将到期 / 已逾期展示能力
- 前后端接口与页面联动验证

### Out of Scope

- 完整登录鉴权 / SSO / RBAC
- 邮件、短信、Webhook 等真实通知通道
- 第三方 IM / 日历 集成
- 大规模权限矩阵与团队空间设计
- AI 提示词系统化配置

## 4. 方案对比

### 方案 A：协作能力增强（推荐）

围绕分享、搜索、筛选、提醒做增量迭代。优点是用户价值直接、风险低、可复用现有 meeting / task 数据模型；缺点是分享安全性先采用轻量方案，后续仍需补鉴权升级。

### 方案 B：先做登录鉴权

优点是长期架构更完整；缺点是会牵动全部接口、前端路由、状态管理与测试，短期交付效率低，不适合作为当前版本的最优切入点。

### 方案 C：继续增强 AI 结果编辑与模板化

优点是能提升纪要质量；缺点是无法解决当前“结果出来后如何被团队消费和跟进”的核心使用问题。

**结论**：优先采用方案 A。

## 5. 用户故事

### 5.1 团队管理者

- 我希望把会议结果发给其他成员查看，而不要求他们先进入系统复杂操作。
- 我希望能快速看到哪些任务快到期、哪些任务已经逾期。

### 5.2 项目经理

- 我希望通过搜索和筛选快速定位某次会议或某类任务。
- 我希望在任务中心按会议维度查看行动项，减少手工翻找。

### 5.3 普通成员

- 我希望通过分享页快速查看会议摘要、转写与任务，而不是进入后台表格式界面。

## 6. 功能设计

### 6.1 会议分享

#### 功能说明

- 在会议详情页增加“生成分享链接 / 复制分享链接”入口
- 每个会议可拥有一个分享令牌
- 分享链接访问公开只读页，展示：会议标题、描述、摘要、转写记录、任务列表

#### 交互规则

- 若会议尚未生成分享令牌，则首次点击时创建
- 后续点击优先复用已有令牌
- `POST /api/v1/meetings/{id}/share` 为幂等接口；若已有分享令牌，则直接返回现有结果，不重复生成
- 每个会议在 V0.2 仅允许一个 active `share_token`
- 分享页不提供编辑、上传、后处理等写操作

#### 安全边界

- 本阶段采用“知道链接即可访问”的轻量模式
- 不在分享页暴露敏感后台操作入口
- 后续如引入登录鉴权，可将公开分享页升级为受控访问
- V0.2 不提供 rotate / revoke / expire 能力，后续若安全要求提升再单独设计
- 分享页仅返回公开展示所需字段，不暴露后台写操作入口或内部存储信息

### 6.2 会议搜索 / 筛选

#### 功能说明

- Dashboard 增加关键词搜索框
- 支持按会议状态筛选（如 scheduled / processing / completed）
- 前端先提供基础筛选 UI，后端接口支持 query 参数过滤

#### 行为规则

- 关键词默认匹配标题，必要时可扩展到描述
- 多筛选条件组合时采用 AND 逻辑
- 无结果时给出空状态提示

### 6.3 任务搜索 / 筛选

#### 功能说明

- Tasks 页增加筛选栏
- 支持按任务状态、优先级、会议 ID 或会议标题筛选
- 任务表格增加到期状态展示列

#### 行为规则

- 状态、优先级支持单选或全部
- 后端任务筛选只支持 `status`、`priority`、`meeting_id`、`keyword`
- 其中 `keyword` 仅匹配任务自身字段，不匹配会议标题
- 前端若提供“按会议标题筛选”，仅作为选择器展示，最终转换为 `meeting_id` 请求后端
- 现有 `assignee_id` 过滤能力保持兼容，不在本版移除
- 筛选变化后刷新当前列表，不改动任务状态更新主流程

### 6.4 基础提醒感知

#### 功能说明

- 不接真实通知通道
- 通过字段计算和 UI 视觉突出实现“提醒感”

#### 展示规则

- 截止时间统一使用单一 datetime 语义；若当前模型无可复用字段，则新增 nullable `due_at`
- 接口统一传输 ISO 8601 datetime；提醒计算使用统一时区基准，后端固定 UTC、前端本地化展示
- `is_overdue = true`：`due_at < now` 且任务未完成
- `is_due_soon = true`：`now <= due_at < now + 48h` 且任务未完成
- `due_at` 为空时，`is_overdue = false`、`is_due_soon = false`
- 已完成任务不显示逾期 / 即将到期提醒

#### UI 建议

- 已逾期：红色 danger tag
- 即将到期：橙色 warning tag
- 正常：灰色或不展示

## 7. 后端设计

### 7.1 数据模型

建议在会议表增加：

- `share_token`：字符串，唯一，可空
- `shared_at`：时间戳，可空

数据库需要为 `share_token` 建唯一索引；历史数据默认 `NULL`，无需回填。

任务截止时间统一以一个字段表达。若当前已有可复用的 datetime 截止时间字段，则直接复用；若当前没有，则本阶段新增 nullable `due_at`，避免同时维护 `due_date` / `due_at` 两套语义。

提醒状态字段 `is_overdue` / `is_due_soon` 不落库，采用响应层动态计算。

### 7.2 API 设计

建议新增 / 调整以下接口：

#### Meetings

- `GET /api/v1/meetings?keyword=&status=`
  - 支持会议列表筛选
- `POST /api/v1/meetings/{id}/share`
  - 创建或返回已有分享令牌
  - `200 OK`
  - 返回：`{ meeting_id, share_token, share_path, created_now }`
  - `share_path` 固定为 `/share/{token}`，前端使用当前 `origin` 拼接完整分享链接
- `GET /api/v1/share/{token}`
  - 返回只读会议详情、转写、任务
  - 仅返回公开展示所需字段
  - 转写数据直接复用现有转写记录，不新增“转写片段提炼”算法
  - 任务仅返回展示字段，如标题、状态、优先级、截止时间、提醒状态，以及可选负责人展示名

#### Tasks

- `GET /api/v1/tasks?status=&priority=&meeting_id=&keyword=&assignee_id=`
  - 支持任务筛选
  - 返回任务时附带提醒状态字段

过滤值严格复用现有 schema / enum；非法值返回 `422`。V0.2 不引入新的分页 / 排序协议，仅做现有列表接口的增量筛选扩展。

### 7.3 服务层职责

- `meeting_service.py`
  - 负责生成 / 复用分享令牌
  - 负责带筛选条件的会议查询
  - 负责公开分享页所需聚合查询
- `task_service.py` 或现有任务服务位置
  - 负责筛选条件解析
  - 负责任务提醒状态计算

### 7.4 错误处理

- 会议不存在：404
- 分享令牌无效：404
- 非法筛选参数：422
- 服务层异常：延续当前统一结构化错误响应规范

## 8. 前端设计

### 8.1 DashboardView

- 增加搜索输入框
- 增加状态筛选控件
- 保留当前卡片式会议列表
- 无结果时展示空态，而不是空白区域

### 8.2 TasksView

- 顶部增加筛选区：状态、优先级、会议、关键词
- 表格增加提醒状态列 / Tag
- 保留现有状态更新交互，避免一次性重构页面

### 8.3 MeetingDetailView

- 增加分享操作按钮
- 成功后可直接复制完整链接
- 若会议尚无摘要 / 转写 / 任务，分享页仍可访问，但按现状展示空态

### 8.4 SharedMeetingView（新增）

- 路由独立，如 `/share/:token`
- 只读呈现会议信息
- 风格偏简洁阅读页，而非后台管理页

## 9. 数据流

1. 用户进入 Dashboard / Tasks 页面
2. 前端携带筛选参数请求后端列表接口
3. 后端服务层执行过滤并返回结果
4. 用户在会议详情页点击分享
5. 前端调用分享接口，得到 token 与 share URL
6. 外部用户访问 `/share/:token`
7. 前端请求公开分享接口，渲染只读页面

## 10. 测试策略

### 后端

- 会议筛选接口测试
- 任务筛选接口测试
- 分享令牌生成 / 复用测试
- 无效 token 404 回归测试
- 任务到期状态计算测试
- 分享接口重复调用返回同一 token
- 无截止时间任务不显示提醒
- 已完成任务不显示提醒
- 公开分享接口不返回内部字段
- 组合筛选按 AND 逻辑生效

### 前端

- 搜索 / 筛选触发正确 API 请求
- 分享按钮成功复制链接或显示结果
- 分享页正确渲染公开数据
- 任务提醒 Tag 渲染正确
- 无结果时展示空态，而非空白页面

### 手动验证

- 创建会议并生成分享链接
- 新开浏览器访问分享页
- 在 Dashboard 输入关键词并筛选
- 在 Tasks 页面组合筛选并检查结果
- 构造即将到期 / 已逾期任务并验证显示
- 构造无截止时间和已完成任务，确认不会误显示提醒

## 11. 验收标准

- 可以为任意会议生成并访问分享链接
- 分享页只读可用，能展示会议核心信息
- 会议列表支持关键词 / 状态筛选
- 任务列表支持多条件筛选
- 任务能明确显示即将到期 / 已逾期状态
- 组合筛选使用 AND 逻辑
- 无结果时接口返回 `200 + []`，前端展示空态
- 后端测试通过，前端 typecheck / build 通过

## 12. 风险与缓解

### 风险 1：公开分享带来安全顾虑

缓解：首期只分享会议基础结果页，不暴露写接口；后续在鉴权版本升级访问控制。

### 风险 2：筛选逻辑扩散到前端和后端两侧

缓解：以后端查询参数为主，前端只负责表单状态与展示。

### 风险 3：任务截止时间字段现状不明确

缓解：实现前先确认现有模型 / schema / migration；若缺失则做最小字段补充，不扩展复杂提醒系统。

## 13. 实施顺序建议

1. 先确认任务截止时间字段现状
2. 后端补会议分享与筛选接口
3. 前端补搜索 / 筛选 / 分享页
4. 增加提醒状态展示
5. 统一做测试、构建、手动联调

## 14. 本阶段不做的事

- 不引入登录注册系统
- 不做团队空间与复杂权限
- 不做真实通知发送
- 不做 AI 能力大幅改造

## 15. 结论

V0.2 最合理的推进方式，是在现有 MVP 基础上补齐“可分享、可查找、可跟进”的协作能力。该方向与立项书中的版本路线一致，能在保持改动可控的前提下，显著提升产品可用性与展示完整度。
