# SmartMeeting 大项目执行总规划（2026-03-17）

## 1. 目标与范围

- **产品目标**：交付可稳定运行的 AI 会议系统 MVP，覆盖会议管理、音频转写、纪要总结、任务提取与任务流转。
- **技术范围**：`backend/`（FastAPI）、`frontend/`（Vue3+TS）、`database/`（SQL 迁移体系）、`docs/`（交付文档）、`scripts/`（运维脚本）。
- **边界约束**：
  - 后端坚持“路由薄、服务厚”。
  - 前端坚持 `<script setup lang="ts">` + Pinia。
  - DB 变更必须遵循 `migrations + seeds + rollback`。

## 2. 分阶段路线图

### Phase A：稳定性与一致性（1 周）

1. 对齐 AGENTS 层级规则（root + 子域）并固化约束。
2. 清理接口入参与错误码一致性（后端）。
3. 补齐前端关键流程的异常可见反馈。
4. 执行一次全链路冒烟（创建会议→上传音频→转写→总结→任务流转）。

### Phase B：业务能力完善（1-2 周）

1. 会议列表与详情能力增强（过滤、排序、分页、状态联动）。
2. 任务中心与用户管理场景完善（筛选、状态流转、指派可用性）。
3. 参与人/转写等关联资源的完整性校验加强。
4. API 文档和前端 runbook 同步更新。

### Phase C：质量与性能（1 周）

1. 后端单测补齐核心服务层路径（meeting/task/audio/llm）。
2. 前端 typecheck/build 常态化（PR 前必过）。
3. 数据库索引和查询路径复核（`EXPLAIN` + `check_performance.sql`）。
4. 关键日志与告警点梳理（上传失败、转写失败、LLM 调用失败）。

### Phase D：上线准备（0.5-1 周）

1. 环境配置模板与敏感信息隔离检查。
2. 回滚预案演练（数据库 + 版本回退）。
3. 发布清单与验收口径冻结。

## 3. 工作流分工建议

## 3.1 后端（backend）

- API 层：参数校验、状态码、错误信息规范化。
- Service 层：会议后处理、任务提取、用户/参与人关系逻辑。
- 基线验证：`pytest` + `ruff` + `mypy`。

## 3.2 前端（frontend）

- 页面层：Dashboard / MeetingDetail / Tasks / Users 连贯体验。
- 状态层：Pinia 中台化状态，不在页面复制业务状态。
- 基线验证：`npm run typecheck` + `npm run build`。

## 3.3 数据库（database + scripts/db）

- 迁移编号连续、可回滚、可健康检查。
- 高查询频率场景优先做索引评估。
- 基线验证：`run_all.sql` + `check_health.sql` + `check_performance.sql`。

## 3.4 文档（docs）

- API 文档、运行手册、变更日志保持同步。
- 每个阶段输出“完成定义（DoD）+ 验证证据”。

## 4. Git 协作与分支策略

- `main`：受保护分支，仅接收通过验证的变更。
- 功能开发：`feat/*`，缺陷修复：`fix/*`，文档：`docs/*`。
- 提交流程：
  1. 小步提交，按主题拆分（避免巨型提交）。
  2. 本地通过必要验证后再 push。
  3. PR 描述包含改动点、影响范围、验证方式、回滚方案。

## 5. 完成定义（DoD）

- 功能：需求验收项全部通过。
- 质量：相关测试通过，类型检查通过，构建通过。
- 数据：迁移可执行、可回滚、健康检查通过。
- 文档：使用方式、配置项、限制与风险记录完整。

## 6. 近期执行清单（可直接开工）

1. 完成分层 AGENTS 文档（已完成）。
2. 按阶段 A 的冒烟链路建立固定检查脚本。
3. 对会议后处理与任务提取路径补回归测试。
4. 在前端新增错误反馈统一组件/策略。
5. 完成一次 DB 性能检查并记录基线。
