# SmartMeeting 核心技术细节与架构复盘

这份文档算是我自己对这个课程项目的技术总结和复盘。一方面是为了答辩时如果老师问到技术细节我能答得上来，另一方面也是给以后的自己留个底，毕竟里面填了不少坑。

## 一、 整体架构思路

一开始我是打算用传统的三层架构写个单体应用，但考虑到现在都流行前后端分离，而且还要接大模型和语音转写，所以最后选了前后端完全分离的 SaaS 架构。
- **前端**负责界面展示和录音采集。
- **后端**负责业务逻辑、鉴权，以及最耗时的语音转写 (Whisper) 和大模型调用。
- **数据库**用了 MySQL，所有的建表、回滚和初始数据填充我都写了原始的 SQL 脚本，通过 Python 脚本来统一下发，这样即使换个环境也能一键还原数据库状态。

部署为了方便交作业，我写了 Docker Compose，不用再去配半天环境。

## 二、 前端部分 (Vue 3 + TypeScript)

前端这块我用的都是目前比较主流的技术栈，主要为了熟练一下 Composition API。

- **框架选型**：Vue 3 + Vite。Vite 打包真的比 Webpack 快很多，平时写代码保存就看到效果了。
- **状态管理**：Pinia。摒弃了老旧的 Vuex，用了 Pinia，我拆分了 `authStore`（存 Token、用户信息）和 `meetingStore`（管理当前会议的状态、摘要、任务）。
- **组件库**：Element Plus。用它搭后台管理界面和数据列表非常方便，页面响应式和排版基本靠它。
- **核心踩坑 - 录音与音频上传**：前端其实有个比较麻烦的点是调用浏览器的 `MediaRecorder` API。由于安全性限制，这玩意必须在 HTTPS 或者 `localhost` 下才能获取到麦克风权限。录制出来的格式可能是 webm 或者 ogg，后端得能处理好。
- **请求封装**：用 Axios 封装了一个 `apiClient`，加了请求拦截器，在每次发请求前把 JWT token 塞进 Request Header 里，如果后端报 401 就会自动跳转到登录页。

## 三、 后端部分 (FastAPI + Python 3.14)

后端是用 Python 写的，选 FastAPI 是因为它天生支持异步，而且有自动生成 Swagger 接口文档的功能，省了写接口文档的时间。

### 1. 核心技术栈
- **框架**：FastAPI + Uvicorn。
- **ORM**：SQLAlchemy 2.0，配合 PyMySQL 连数据库。
- **依赖管理**：纯原生的 `requirements.txt`。其实想过用 Poetry，但为了交作业环境越简单越好。
- **鉴权机制**：JWT (JSON Web Token)。登录成功后发一个 Token，后续访问只要带上这个 Token 就行。

### 2. AI 集成（最大的挑战）
这部分是我花时间最多的地方，为了让应用不管在什么机器上都能跑起来，我做了非常多的“回退”（Fallback）处理。

#### 语音转写 (ASR) 与音频预处理
- **音频预处理 (FFmpeg)**：前端浏览器通过 `MediaRecorder` 传上来的录音流，格式通常是 `webm` 或 `ogg`。但大模型对输入音频的采样率（通常要求 16kHz）和格式有比较严格的限制。因此，后端必须依赖系统级的 `FFmpeg` 工具。我在接收到音频流后，通过后端的临时文件和 Python subprocess / audio 库去调用 FFmpeg，将音频统一重采样并转码为标准的 WAV/MP3 格式，再喂给模型。这也就要求部署环境中（无论是物理机还是 Docker 容器）必须提前安装好 FFmpeg。
- **架构选型 (Faster-Whisper 胜出)**：起初我在测试时集成的是 OpenAI 官方基于原生 PyTorch 架构跑的 `openai-whisper`。但在纯 CPU 环境下慢得令人发指（转写时间经常比录音本身还长）。为了优化性能，我最终在生产/部署环境中把核心转写引擎切到了 `faster-whisper`。它是基于 CTranslate2 引擎重写的，量化（Quantization）做得更好，内存占用大幅缩小，推理速度比原生 PyTorch 版快上好几倍。我在代码里依然保留了两套接口，但默认主力使用的是 `faster-whisper` 的 `base` 甚至 `small` 模型来平衡速度和准确率。
- **硬件加速动态切换 (CUDA vs CPU)**：虽然 `faster-whisper` 在 CPU 上已经有了很大改善，但为了彻底解放性能，系统依然需要 PyTorch 及其对应的 CUDA 运行时的支持。我在后端的初始化代码里写了自动检测逻辑（类似于查验 `torch.cuda.is_available()` 或让 faster-whisper 尝试获取 `cuda` device）。如果检测到用户的机器装了 N 卡且配置好了 CUDA，就会自动开启 GPU 加速，转写速度能实现“秒出”；如果是在一般的轻薄本上跑，就会无缝平滑降级到 `cpu` 运行。
- **环境兜底 (Mock ASR)**：开发和答辩最怕的就是“掉链子”。如果机器配置实在太差，连 Python 和 FFmpeg 环境都装不全跑不起来模型怎么办？我还专门写了个 `Mock` 机制。只要在 `.env` 里配置降级，它就会跳过真实的音频解析和模型加载，而是根据音频长度等信息，假装转写成功，直接吐出一段预设的中文会议记录。这就确保了不管服务器算力多拉胯，我的演示流程（录音->转写->摘要）一定能跑通。

#### 大语言模型 (LLM)
- **多模型支持**：会议转写出来的文本比较杂乱，所以我接了 LLM 来生成摘要和提取代办任务。我写了一个统一的 LLM Service，底层既支持调官方的 OpenAI API（比如 GPT-4o-mini），也支持对接到本地的 Ollama（如果自己有算力的话）。
- **规则兜底 (Rule Fallback)**：万一 API Key 欠费了，或者答辩教室没网断连了怎么办？我写了一套备用的规则引擎兜底。如果没有可用的 LLM 资源，系统会通过正则匹配和关键词提取的土办法，强行截取会议的前几句作为摘要，从而不至于让程序崩溃。

## 四、 数据库设计

用的是 MySQL 8.0。主要的表有：
- **users**: 存账号、加盐哈希后的密码。
- **meetings**: 存会议的主体信息（标题、时间、状态、音频路径、摘要文本）。
- **tasks**: 存从会议里提取出的代办任务，并通过外键关联到 users（创建人/执行人）和 meetings（所属会议）。

为了保证开发时的规范，我没用 Alembic 去做自动迁移，而是老老实实写了 `migrations/` 下的 SQL 文件，再配上 `scripts/db/run_all.sql` 来按顺序执行。另外还有个 `check_health.sql` 来验证表结构。

## 五、 测试与工程化

作为一门完整的课程作业，我不能光有代码不测试。
- **后端测试**：用了 Pytest。写了 161 个测试用例。这里的坑是，跑测试不能弄脏开发库，所以我专门用 fixtures 给测试环境 override 了一个临时的 SQLite/测试库。
- **前端测试**：用了 Vitest。主要是测了几个核心组件和 Store 的逻辑。
- **一键脚本**：写了个 `scripts/dev/qa.py`。这个脚本会帮我把数据库顺序检查、后端测试、前端类型检查 (`vue-tsc`) 统统跑一遍，算是个低配版的 CI/CD 流水线，每次改完代码跑一下这个，心里就有底了。

## 六、 总结

这个项目麻雀虽小五脏俱全。看似是个简单的增删改查系统，但加了音频处理、AI 接口调度、异步任务这些东西之后，链路其实挺长的。尤其是各种 Mock 和 Fallback 机制的加入，让这个系统不仅是个“能看”的 Demo，而是一个“在哪儿都能跑得起来”的系统。这应该是我在这个项目里学到最实用的经验。
