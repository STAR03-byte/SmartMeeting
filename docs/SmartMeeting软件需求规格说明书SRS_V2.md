# 1 引言

## 1.1 项目背景

SmartMeeting 是一款 AI 驱动的轻量化会议 SaaS 产品，面向 5-50 人规模中小团队、远程协作团队和小型项目组。产品核心目标是替代人工会议记录，打通“会议记录→智能纪要生成→任务拆解分配→跟进督办提醒”全流程闭环，解决会议记录低效、会后执行断层、任务无人落实的核心痛点。

本版本在 V1.0 需求基线基础上进行正式修订，结合当前产品定位与已实现能力，对在线录音、实时语音转写、纪要分发、任务督办、远程协作与轻量化商业约束进行补充，形成可直接用于设计、开发与验收的正式需求规格。

## 1.2 编写目的

本文档旨在：

1. 将产品目标转化为可验证的软件需求规格；
2. 明确各功能模块的输入、处理逻辑、输出与业务规则；
3. 定义非功能性需求、接口需求与验收标准；
4. 为后续开发、测试、部署与交付提供统一依据。

# 2 总体描述

## 2.1 产品愿景

SmartMeeting 致力于成为中小团队“会议到执行”的轻量化闭环工具。产品愿景是让每一次会议的结论都能自动结构化沉淀为可执行的任务，让团队协作从“开了会”变成“做成了”。

## 2.2 用户特征

|用户角色|典型场景|核心诉求|技术水平|
|--------|--------|--------|--------|
|团队管理者|主持周例会、项目评审|快速把控会议结论，明确任务责任与时限|中等，熟悉 Web 应用|
|项目经理|项目进度会、需求讨论会|拆解会议任务、分配到人、跟踪状态|中等偏高|
|普通参会成员|日常站会、技术讨论|回顾会议重点，知晓个人任务|一般|
|系统维护者|部署、验收、日常维护|保持系统可用、可测试、可回退|中等|

## 2.4 主要约束

1. **开发约束**：只保留高频刚需模块，禁止开发复杂高阶功能。
2. **操作约束**：流程极简，尽量零培训上手，无深层折叠菜单。
3. **质量约束**：优先保障基础功能稳定性，再优化体验细节。
4. **性能约束**：兼容弱网与普通设备，加载速度快，无明显卡顿。
5. **AI 约束**：复用成熟轻量化 AI 能力，禁止过度自研算法。
6. **缓存约束**：单场会议离线缓存 ≤ 10M。

## 2.5 假设

1. 用户拥有基本的 Web 应用操作能力。
2. 用户可使用电脑或手机自带麦克风进行录音。
3. 用户的音频文件为常见格式（WAV、MP3、M4A 等）。
4. 服务器可访问互联网以调用 AI 服务，或已配置本地模型服务。
5. MySQL 数据库已正确部署并可连接。

## 2.6 产品范围

### 2.6.1 V2 必做范围

- 在线录音与实时语音转写（含 VAD、说话人识别、热词增强、标点恢复）
- 文件导入转写
- 智能纪要生成
- 纪要分发与导出
- 任务提取与督办
- 用户、会议、参与者、任务基础管理
- Web 端与微信小程序入口

### 2.6.2 非首版范围

- 系统音采集
- 多轨录音
- 高清视频会议
- 企业级 SSO / LDAP 作为主路径
- 复杂 BI 与多层级任务拆分

## 2.7 商业与版本约束

|版本|用户规模|会议配额|时长限制|说明|
|----|--------|--------|--------|----|
|免费版|5 人|每月 20 场|单场 ≤ 60 分钟|基础转写、简易纪要、基础任务功能|
|付费版（小）|5-20 人|按套餐|按套餐|支持按月试用|
|付费版（中）|21-50 人|按套餐|按套餐|支持按年付费|

- 不限制登录设备数量。
- 不设置隐藏收费。

# 3 具体需求

## 3.1 功能需求

### 3.1.1 用户认证模块（FR-AUTH）

#### 3.1.1.1 用户登录（FR-AUTH-01）

|项目|描述|
|----|-----|
|功能描述|用户通过用户名和密码登录系统，获取认证令牌|
|输入|用户名、密码|
|处理逻辑|验证用户名与密码，生成 JWT access_token|
|输出|`{access_token: string, token_type: "bearer"}`|
|业务规则|用户名唯一；token 过期后需重新登录|
|异常处理|用户名或密码错误 → 返回 401|

#### 3.1.1.2 获取当前用户（FR-AUTH-02）

|项目|描述|
|----|-----|
|功能描述|通过 token 获取当前登录用户信息|
|输入|Authorization: Bearer token|
|处理逻辑|验证 token 并查询用户信息|
|输出|用户对象|
|异常处理|token 无效 → 返回 401|

### 3.3.3 接口错误码规范

|HTTP 状态码|含义|使用场景|
|-----------|-----|---------|
|200|成功|GET/PATCH 请求成功|
|201|已创建|POST 请求成功创建资源|
|204|无内容|DELETE 请求成功|
|400|请求错误|缺少必要参数、逻辑错误、无音频、无转写|
|401|未认证|token 缺失或无效|
|404|未找到|资源不存在|
|422|验证错误|请求参数格式错误、枚举值非法|
|500|服务器错误|未预期异常或服务不可用且无降级路径|

错误响应格式：
```json
{
  "detail": "错误描述信息",
  "error_code": "ERROR_CODE_NAME"
}
```

# 4 其他需求

## 4.1 数据管理需求

### 4.1.1 数据持久化

|数据类型|存储方式|备份策略|
|--------|--------|---------|
|业务数据（用户/会议/任务等）|MySQL 8.0|定期备份|
|音频文件|本地文件系统或云端存储|定期备份|
|配置信息|.env 文件|不纳入版本控制|

### 4.1.2 数据保留策略

- 用户数据：账号删除后按业务规则处理关联资源。
- 音频文件与转写：随会议留存策略处理。
- 任务数据：按会议归档，支持追溯。
- 纪要数据：支持留存时长配置，到期自动清理。

### 4.1.3 数据库迁移

- 采用 SQL 脚本驱动的迁移方式（非 alembic）。
- 迁移脚本编号执行，提供回滚脚本。
- 种子数据可重复执行。

## 4.2 故障处理需求

### 4.2.1 故障检测

|故障类型|检测方式|处理方式|
|--------|--------|---------|
|数据库连接失败|启动时检查|开发环境可降级；生产环境报错退出|
|Whisper 不可用|导入/运行失败|回退到 mock ASR|
|LLM 不可用|API key 缺失或超时|回退到规则生成|
|麦克风不可用|浏览器权限或能力不足|提示用户并停止流程|
|文件导出失败|导出接口异常|返回明确错误并保留源数据|

### 4.2.2 错误日志

- 后端使用日志模块记录关键链路。
- 关键操作包括登录、录音、转写、后处理、分发、导出。
- 异常堆栈可追踪，便于排查。

## 5.2 典型用例

### 核心用户用例

1. **登录系统**：输入用户名密码 → 系统验证 → 跳转仪表盘。
2. **录制会议**：进入会议详情页 → 点击开始录音 → 结束后自动转写并刷新内容。
3. **生成纪要**：会议结束后触发后处理 → 输出结构化纪要与任务。

5. **分发纪要**：一键导出/外链分享/推送至办公平台。
