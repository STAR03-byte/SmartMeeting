# SmartMeeting 智能会议系统 - 项目立项规划书

## 一、立项摘要

SmartMeeting 是一款面向中小企业与远程协作团队的智能会议 SaaS 产品。项目通过 AI 语音转写、纪要生成与任务提取能力，解决会议记录效率低、会后执行断层、跨团队协同弱等问题，目标在 10 周内交付可上线的 MVP，并在 3 个月内完成首批种子用户验证。

---

## 二、项目背景与机会

### 2.1 业务背景

- 企业线上会议频次持续增长，但会议信息沉淀和执行闭环能力不足。
- 人工记录成本高且质量不稳定，关键结论和行动项容易遗漏。
- 现有工具多聚焦“记录”而非“执行”，缺少从会议到任务落地的一体化链路。

### 2.2 市场机会

- AI 大模型与语音识别能力成熟，会议场景自动化可行性显著提升。
- 中小团队对“轻量、低门槛、可快速上线”的 SaaS 工具有明确需求。
- 通过“转写 + 纪要 + 任务”的高频刚需切入，可形成稳定使用习惯与续费基础。

---

## 三、项目目标与成功指标

### 3.1 项目目标

1. 构建可商用的会议智能化 MVP，覆盖会前、会中、会后核心流程。
2. 将会议纪要产出效率提升 70% 以上。
3. 将会议任务可追踪率提升至 90% 以上。

### 3.2 成功指标 (KPI)

| 维度 | 指标 | 目标值 | 统计周期 |
|------|------|--------|----------|
| 效率 | 单次会议纪要生成时间 | <= 3 分钟 | 每周 |
| 质量 | 语音转写准确率 | >= 95% (普通话场景) | 每周 |
| 执行 | 任务提取准确率 | >= 85% | 每周 |
| 使用 | 周活跃团队数 (WAU Team) | >= 20 (MVP 后第 4 周) | 每周 |
| 留存 | 7 日留存率 | >= 35% | 每周 |

## 四、产品定位与用户画像

### 4.1 产品定位

| 属性 | 说明 |
|------|------|
| 产品形态 | SaaS Web 应用 |
| 目标用户 | 中小企业、远程团队、项目型协作组织 |
| 核心价值 | 让会议结论自动结构化，让任务执行可追踪 |
| 差异化 | AI 驱动的“会议到执行”闭环，而非仅记录 |

### 4.2 核心用户画像

| 角色 | 典型诉求 | 关键痛点 |
|------|----------|----------|
| 团队管理者 | 快速掌握会议结论与负责人 | 会后信息分散、推进成本高 |
| 项目经理 | 把纪要转为任务并跟踪状态 | 手工拆解任务耗时、遗漏多 |
| 普通参会成员 | 低成本回顾会议重点 | 信息过载、难定位关键决策 |

## 五、产品范围定义

### 5.1 MVP 范围 (In Scope)

- 会议音视频上传与语音转写
- 结构化会议纪要自动生成
- 决策与待办事项自动提取
- 任务分配、状态流转、到期提醒
- 会议记录共享与基础权限管理

### 5.2 非 MVP 范围 (Out of Scope)

- 深度视频会议 SDK 集成
- 企业级 SSO/LDAP 集成
- 复杂 BI 报表与高级分析
- 私有化本地部署版本

## 六、功能规划

### 6.1 版本路线图

| 版本 | 时间 | 核心内容 |
|------|------|----------|
| V0.1 (MVP) | 第 1-10 周 | 转写、纪要、任务提取、任务管理 |
| V0.2 | MVP 后 4-6 周 | 团队协作增强、模板化纪要、权限细化 |
| V0.3 | MVP 后 8-12 周 | 统计分析、第三方日历/IM 对接 |

### 6.2 核心功能优先级

| 优先级 | 模块 | 功能点 | 价值说明 |
|--------|------|--------|----------|
| P0 | 语音转写 | 上传转写、实时转写、多语言识别 | 建立会议数据基础 |
| P0 | 智能纪要 | 自动摘要、结构化模板、重点标注 | 降低整理成本 |
| P0 | 决策提取 | 决策识别、任务抽取、负责人识别 | 形成执行闭环 |
| P1 | 任务管理 | 状态流转、提醒、看板视图 | 提升落地效率 |
| P2 | 团队协作 | 分享、评论、权限控制 | 支持组织化协作 |

## 七、技术方案

### 7.1 技术栈

| 层级 | 技术选型 |
|------|----------|
| 前端 | Vue 3 + TypeScript + Element Plus |
| 后端 | FastAPI (Python) |
| 数据库 | MySQL |
| AI 能力 | Whisper + LLM |
| 部署 | Docker + Nginx |

### 7.2 系统架构

```text
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│   Frontend   │───▶│   FastAPI    │───▶│    MySQL     │
│  Vue3 + TS   │    │  API/Service │    │   Storage    │
└──────────────┘    └──────┬───────┘    └──────────────┘
                           │
                  ┌────────┴────────┐
                  │  AI Pipeline    │
                  │ Whisper + LLM   │
                  └─────────────────┘
```

### 7.3 非功能要求

- 可用性: 核心 API 可用性 >= 99.5%
- 性能: 30 分钟会议在 3-5 分钟内完成纪要产出
- 安全: 传输全程 HTTPS，敏感配置使用环境变量管理
- 可观测性: 关键链路日志、错误告警、调用耗时监控

## 十、风险评估与应对

| 风险 | 影响 | 概率 | 应对策略 |
|------|------|------|----------|
| AI 成本波动 | 运营成本上升 | 中 | 增加模型路由策略与缓存复用 |
| 转写质量不稳定 | 用户体验下降 | 中 | 场景化调优、关键词纠错、人工校对入口 |
| 单人开发瓶颈 | 交付延迟 | 中高 | 严控范围，优先 P0，减少并行任务 |
| 并发增长导致性能问题 | 稳定性下降 | 中 | 提前压测，关键服务可水平扩展 |

## 十二、验收标准

MVP 上线验收需同时满足以下条件:

1. P0 功能全部可用并通过核心流程测试。
2. 关键接口稳定性满足可用性目标。
3. 至少完成 20 场真实会议样本验证。
4. 已形成部署文档、运维手册和问题排查清单。
